# Engineering Logs & Guardrails

To prevent repeating the same bugs and architectural mistakes, here is the list of identified anti-patterns and engineering rules derived from past fixes.

## 🛠️ The Trading System Engineering Playbook

### 1. Market Calendar vs. Gregorian Calendar
*   **The Anti-Pattern**: Advancing historical backtests or expiry dates using calendar-day logic (`timedelta(days=1)`) or standard weekday checks. This skips market holidays (e.g. Diwali, NSE holidays) and introduces weekend skew/Monday lock-in.
*   **The Rule**: Never use raw day-arithmetic for trade/backtest advancement or expiration checks. Always use calendar utilities (`next_trading_day`, `count_trading_days`) backed by actual exchange calendars.

### 2. State & Filter Desynchronization
*   **The Anti-Pattern**: Modifying signal scores or directions in post-processing filters (FII flows, concentration limits, macro events) without atomically updating confidence, counts, stop-loss/targets, or trade plan profiles.
*   **The Rule**: Keep base technical signals and risk-mitigation filters decoupled. Run assessment/trade-plan recalculation as a single, atomic post-filter step (`_recompute_assessment_and_trade_plan()`) so that downstream systems (cron, database caching, UI conviction badges) never use mismatched or pre-filter metrics. Ensure that any multi-pass assessment (e.g. running a second pass to compute Kelly sizing using the updated risk-reward ratio) always calls `_update_trade_plan()` at the very end of the final pass to guarantee that the direction and trade plan fields are perfectly synchronized. Additionally, if a stock is demoted to `NEUTRAL` post-filtering, it must be excluded from the final recommendation results and the `total_with_signals` count to prevent count mismatch discrepancies in user-facing dashboards.


### 3. Data-Tier Safety & Sanitization
*   **The Anti-Pattern**: Duplicate trade insertion across paper/shadow caches, `ZeroDivisionError` on strategies with perfect win/loss records (no losses), and simulations defaulting or crashing on `NULL` database column values.
*   **The Rule**:
    *   Enforce deterministic Python-tier deduplication on composite keys like `(ticker, entry_date, fingerprint)`.
    *   Guard all math formulas (e.g., Kelly Criterion, win rates) and scanners against boundary conditions (zero losses, zero wins, empty datasets, or division by zero on zero average volume). If average volume is 0 but current volume is non-zero, treat average volume as 1 to avoid silent drops of valid breakouts.
    *   Provide explicit safe fallbacks for `NULL` fields in historical records during simulation passes.

### 4. Price-Level Ingestion (Intraday vs. EOD)
*   **The Anti-Pattern**: Using closing prices (`current_close`) for gap detection or backtest triggers, which misses intraday fill-and-reversal behavior and reports incorrect outcomes.
*   **The Rule**: Intraday logic must check extreme levels (`current_high`/`current_low`) rather than EOD closes to determine gap fill/fade status and stop-loss hits.
*   **The Breakout/Breakdown Exception**: Unlike gap-fills and stop-loss execution, breakout and breakdown triggers must be confirmed on a closing price basis (e.g. `current_close > n_day_high` or `current_close < n_day_low`) to avoid triggering false bullish/bearish signals on intraday spikes that fail to hold, which also aligns the triggers with the backtest entry price (`today_close`).

### 5. Compounding & Drawdown Realism
*   **The Anti-Pattern**: Understating drawdowns by using static cash sizes, neglecting marked-to-market unrealized P&L, sorting entry after exit on same-day trades, or omitting hit stop-losses.
*   **The Rule**: Simulations must emulate exact exchange logistics: sort same-day trades (entry before exit), include live mark-to-market unrealized P&L, handle stop-losses, and fetch fresh quotes for active positions.

### 6. Query Hygiene & Performance
*   **The Anti-Pattern**: Executing queries in loops (N+1 query problem) when fetching trade stats, causing performance degradation under load, and swallowing exceptions silently.
*   **The Rule**: Batch database operations using `IN` clauses, implement in-memory caching for repetitive computations, and always use structured logging to surface errors rather than swallowing them.

### 7. Technical Indicator Fidelity (Volatility Calculations)
*   **The Anti-Pattern**: Mixing indicator definitions or using simple arithmetic means (SMA) where Wilder's smoothing/exponential smoothing is mathematically specified and expected (e.g. for Average True Range). This underestimates short-term volatility spikes, leading to overly tight stop-losses and premature stop-outs.
*   **The Rule**: Ensure mathematical implementations of indicators strictly match the industry standards described in their documentation/docstrings. Volatility measures (such as ATR) must use proper smoothing methods if they serve as inputs for risk-management parameters (stop-losses, target levels, or position sizes).

### 8. Shadow Trade Direction-Awareness
*   **The Anti-Pattern**: Hardcoding long-only P&L calculations (`(price - entry) / entry * 100`) in counterfactual trackers (like shadow trades) because the current active configuration only tracks buys. If short signals are later enabled or recorded, their P&Ls become inverted, poisoning the ML calibration datasets with corrupted labels.
*   **The Rule**: Always calculate shadow/paper trade P&Ls using a direction-aware multiplier (e.g. `multiplier = 1 if LONG else -1`) to ensure schema columns and ML model retraining logic remain structurally consistent and future-proof.

### 9. System Outage vs. Market Risk Decoupling
*   **The Anti-Pattern**: Treating technical system failures (e.g., recommender scan timeouts or API outages) as standard market caution flags in decision arrays, or modifying final decision outcomes post-hoc. This leads to double-counting cautions and mismatching the reasoning states stored in the database or rendered in the UI (e.g., a "RED / STAND DOWN" verdict displaying a low caution flag count).
*   **The Rule**: Keep technical system health overrides completely decoupled from market risk evaluation decision tables. Check system-level failures first as a hard short-circuit/override. Do not append technical errors to standard caution/risk flags, ensuring that caution counts, actions, and labels are always atomically aligned and clean for UI rendering and calibration history.

### 10. Avoid Self-Referential (Lookahead) Indicators
*   **The Anti-Pattern**: Including today's candle (or the current tick) in historical lookback arrays when calculating proximity to support/resistance levels or other relative boundary conditions. On breakout or breakdown days, this causes the price itself to redefine the historical high/low, resulting in self-contradicting signals (e.g. flagging a stock as bearish "near major resistance" on the exact day it achieves a bullish breakout).
*   **The Rule**: Always exclude today's candle (slice with `[:-1]` or `[-N-1:-1]`) when establishing historical high/low levels or support/resistance boundaries for proximity checks. Additionally, enforce non-negative constraints (e.g., `0 <= distance < threshold`) to prevent negative distances from triggering the proximity conditions on a breakout or breakdown day.

### 11. Thread-Safe Database & Network Operations (SQLite Concurrency & Timezones)
*   **The Anti-Pattern**: Executing database write operations (like SQLite cache updates) inside parallel worker threads (e.g. `ThreadPoolExecutor`). This leads to concurrent write contention, causing `sqlite3.OperationalError: database is locked` errors. Similarly, making repetitive synchronous API requests inside nested loops (e.g., checking pairwise combinations) poisons execution time and gets the server rate-limited or blocked.
*   **The Rule**: Keep parallel thread pools read-only with respect to the database. Use threads exclusively for non-blocking network fetches (like `yfinance` history). Alignment, calculation, and database writes must be serialized and executed on the main thread. When pre-computing combination metrics, pre-fetch all required raw data once in parallel first, then perform pairwise operations in memory sequentially.
*   **The Rule (Timezones)**: Always use `datetime.utcnow()` instead of `datetime.now()` when checking expiration/staleness of database timestamps stored using SQLite's `datetime('now')` to avoid timezone offset discrepancies (e.g., IST vs UTC).

### 12. Network-Safety & Request-Path Decoupling (No Synchronous Network Fetches & Default Timeouts)
*   **The Anti-Pattern**: Allowing user-facing API endpoints (such as the daily verdict or recommender lists) to trigger synchronous external network requests (like calling `yfinance` history for up to 50 tickers) if the database cache is cold/empty. If even one external request hangs due to network issues or rate limits, the entire ThreadPoolExecutor and FastAPI worker thread block indefinitely because Python joins worker threads on executor shutdown.
*   **The Rule**:
    *   **Decouple Web Request Paths**: User-facing API endpoints must never trigger synchronous network requests on cold cache lookups. Pass a `fetch_if_missing=False` flag down to scanners and recommenders so they immediately return `None` (fail-fast) if the cache is empty, forcing the system override to handle it gracefully in milliseconds.
    *   **Enforce Global/Default Timeouts**: Always set a default socket timeout (e.g., `socket.setdefaulttimeout(15)`) at application startup so that background or cron network threads are guaranteed to timeout and release their connections rather than hanging indefinitely.

## Rejected Changes

- **Baseline Volatility Samples Wrong Historical Windows**: Assumed that the baseline volatility calculation in `backend/market_regime.py::classify_regime_for_date()` samples from random disjoint historical windows and proposed replacing `closes[:-i]` with `closes[-i-20:-i]`. This was rejected because the current code already computes rolling, overlapping 20-day volatilities spaced 5 days apart. Furthermore, the suggested fix slices an array of exactly 20 elements, which fails the internal length check in `_annualized_vol()` (which requires `window + 1` or 21 elements) and would cause it to always return `0.0`, breaking the `HIGH_VOL` regime detector. Even if corrected to 21 elements (`closes[-i-21:-i]`), the result is index-wise identical to the current code.
- **Neutral Signal Direction Bug**: Alleged that treating neutral `"HOLD"` signals with `direction = 1` in the backtest outcome block inflates backtest metrics and contaminates ML training data. This was rejected because overall summary statistics in `run_recommender_backtest` explicitly filter out `"HOLD"` signals, and the ML training dataset only queries from `paper_trades` and `shadow_trades` (which do not track neutral/HOLD signals). Removing `"HOLD"` signals or changing their return/direction would directly violate the rule requiring neutral backtest scores to emit `"HOLD"` signals (to prevent silent nulls in the UI) and would break the test suite.
- **Silent Data Gaps on Recommender Failure**: Alleged that recommender failures (`recommender_failed = True`) cause `snapshot_today()` to fail and leave gaps in `verdict_history`. This was rejected because `compute_daily_verdict()` catches recommender exceptions internally and successfully returns a `"RED"`/`"STAND DOWN"` verdict, which is then persisted normally. The suggested fix of catching exceptions in `snapshot_today()` and appending technical error messages to `caution_flags` would violate the decoupling rules in **Rule 9 (System Outage vs. Market Risk Decoupling)**.
- **Intraday Price Extremes using current_low/current_high**: Alleged that checking `current_low <= prev_close` and `current_high >= prev_close` in `recommender.py` is incorrect because it uses the daily high/low candle fields from `yfinance` history and fails to distinguish between "intraday extremes so far" during market hours vs "EOD close". This was rejected because the daily High and Low candle fields *are* the actual intraday price extremes of the candle. During market hours, they naturally represent the intraday extremes observed so far, and after market close, they represent the final intraday extremes. Using them for gap fill detection is mathematically correct and consistent.
- **Double-Counting Caution Flags When FII/DII is MIXED**: Alleged that when FII bias is BEARISH and DII is absorbing, the daily verdict double-counts caution flags by entering the bearish block first and then the DII offset block to set the bias to MIXED. This was rejected because the DII offset logic is encapsulated inside `backend/fii_dii.py::get_market_bias()`, which resolves the bias status to `"MIXED"` *prior* to returning the bias object. In `backend/daily_verdict.py`, the mutually exclusive `if-elif` chain prevents any bearish caution flags from being appended if the bias is `"MIXED"`, resulting in exactly one caution flag.
- **Filter Adjustments Silently Discarded from Counts and Confidence**: Alleged that `_recompute_assessment_and_trade_plan()` silently discards filter adjustments and fails to update confidence/counts, leaving the UI showing "4 bullish signals, HIGH confidence" for demoted `NEUTRAL` stocks. This was rejected because: (1) `_recompute_assessment_and_trade_plan()` already includes `filter_adjustments` in `combined_signals` for both passes of `get_honest_assessment()`; (2) it is never called in isolation, but is invoked by the four filter wrapper functions which immediately run `_recompute_confidence_and_counts(result, include_filters=True)` and return the updated dict; and (3) setups demoted to `NEUTRAL` are filtered out from API recommendation lists, meaning they are never shown in the UI recommendations at all.
- **L1-Regularized Logistic Regression Convergence Diagnostics & Underfitting**: Alleged that `fit_l1_logistic_regression()` underfits on small feature subsets because it lacks convergence diagnostics and doesn't force a minimum number of non-zero coefficients. This was rejected because: (1) proximal gradient descent (ISTA) with 2000 iterations and `lr=0.1` is mathematically guaranteed to converge for small binary feature sets; (2) out-of-fold cross-validation metrics are already used for promotion decisions, so the model does not evaluate on its own training data; and (3) L1 regularization's primary mathematical purpose is sparse feature selection. Forcing non-zero coefficients when signals are weak would cause overfitting and violate the "Honest Forecasting" principles, whereas falling back to the base rate (intercept/regime coefficients) is the mathematically correct and low-risk behavior for trading.
- **N² Shadow Trade Refresh Bomb**: Alleged that calling `refresh_shadow_prices()` inside the active paper-trade loop in [simulation.py](file:///d:/Code/indian-trading-agent/backend/simulation.py) triggers $O(N \times M)$ redundant network calls. This was rejected because the call to `refresh_shadow_prices()` is already located outside of the `for trade in trades:` loop, executing exactly once after all paper trades are processed.
- **SQLite Writes from ThreadPool Workers in Backtest**: Alleged that `save_recommender_backtest_row(result)` is invoked directly from within the `ThreadPoolExecutor`'s worker thread, triggering concurrent write contention and violating Rule 11. This was rejected because the `as_completed(futures)` iterator is consumed sequentially by the main thread. The worker threads only perform non-blocking network fetches and math computations in `_analyze_stock_at_date()`, while the actual database writes are serialized safely on the main thread, perfectly adhering to Rule 11.
